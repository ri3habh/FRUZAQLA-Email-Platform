Pharmaceutical companies need to create marketing content (emails, banner ads, social posts, slide decks) under strict FDA regulations: [cite: 7]
* Content must come from approved sources. All claims must be sourced from clinical literature or previously approved content-nothing can be invented or paraphrased. [cite: 8]
* Visual assets are restricted. All images, icons, and graphs must come from an approved library. [cite: 9]
* Content goes through heavy iteration. Expect 5-10+ revision cycles before a piece is approved. [cite: 10]
* Everything must pass compliance checks. Every claim is fact-checked, FDA rules verified, and channel compatibility confirmed. [cite: 11]

Target therapeutic: FRUZAQLA [cite: 12]

### Appendix: Domain Workflow Reference [cite: 71]
This section describes the full content creation workflow in more detail. [cite: 72] Use it as context for your design decisions-you are not expected to implement all of these steps. [cite: 73]

1. Request Creation [cite: 74]
Users specify content type (email, banner ad, social post, slide deck), target audience (HCPs, patients, caregivers), campaign goal (awareness, education, CTA), therapeutic context, and tone (clinical, empathetic, urgent, informative). [cite: 75]

2. Conversational Alignment [cite: 76]
The system asks clarifying questions, the user provides context on messaging priorities, the system suggests relevant approved claims, and the user selects which to include and their priority. [cite: 77]

3. Claim Selection & Sourcing [cite: 78]
Every claim must be traceable to its source. Display format example: "This claim sourced from: [FRUZAQLA Phase 3 Trial, The Lancet, 2023]." [cite: 79] Users must explicitly approve each claim before generation proceeds. [cite: 80]

4. Content Generation [cite: 81]
The system assembles HTML output by combining selected claims with formatting, applying brand guidelines, inserting approved visual assets, and maintaining compliance metadata (sources, approval dates). [cite: 82]

5. Iterative Editing [cite: 83]
Two editing modes: natural language ("Move the efficacy data above the safety information") and direct inline editing with change tracking. [cite: 84] Also includes asset swapping from the approved library and automatic compliance re-validation after changes. [cite: 85]

6. Compliance Review [cite: 86]
Before export, the system validates: all claims match the approved library (exact text match), visual assets are from the approved library, FDA-required disclosures are present (ISI, fair balance), grammar and spelling are correct, and channel compatibility is confirmed. [cite: 87] Results are green checks, yellow warnings, or red flags (which block export). [cite: 88]

7. Export & Distribution [cite: 89]
Final output includes: an HTML file with embedded inline CSS, a metadata file (claims used, sources, approval chain), a compliance report, and an asset manifest. [cite: 90]
